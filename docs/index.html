Unified CLI

Native binary built with Bun. Inspect methods, run tests with per-step trace logs, execute SQL queries, and trigger cross-environment migrations. Namespaced under belz ad, belz pd, belz tw, and belz migrate.

Smart Extension

Manifest V3 extension for Chrome/Edge. JSON bulk editors, CodeMirror textarea modals, one-click output copy, and keyboard shortcuts wired directly into the NSM web platform.

Zero Auth Latency

Session files persist Bearer tokens per environment. On 401, the auth module re-logs in automatically and retries the failed request — no manual token management ever.

AI Ready (ACP)

A Next.js web app bridges the Agent Client Protocol over stdio to coding agents (Claude Code, Gemini, Codex, OpenCode). Supports streaming text, tool calls, permission gates, managed terminals, and constrained filesystem access — all in a single singleton Node.js process.

Teamwork Integration

Fetch task details, comments, and status from Teamwork project management. Query by task ID or URL with belz tw task and belz tw comments.

System Architecture

Flat monorepo. Four source modules (no package.json) consumed by one Bun workspace binary. A separate Next.js app handles the AI bridge.

Request Lifecycle

1. Entry — cli.ts or cli-build.ts receives process.argv.
2. Routing — runNamespacedCli() splits on the first arg (ad / pd / tw / migrate) and looks up the command in the registry map.
3. Auth — apiFetch() loads a Bearer token from ~/.belz/sessions/<env>.json; if absent, calls login() first.
4. API — HTTP request to the active NSM environment URL. On 401, session is cleared and retried once.
5. Parse — Raw response is cast into domain types by module-level parsers (e.g., parseMethodResponse()).
6. Render — renderHuman() outputs colored tables; renderLLM() outputs the full CommandEnvelope as JSON.

api.ts — HTTP Client

apiFetch(path, options) is the single HTTP entry point for all modules. Attaches Authorization: Bearer <token>, supports three auth modes (Bearer, Raw, None), and automatically re-authenticates + retries on 401.

// Auth flow inside apiFetch: 1. loadSession() → attach Authorization header 2. fetch(url, { headers }) 3. if (response.status === 401): clearSession(); await login(); retry once

runner.ts — CLI Dispatcher

Two functions handle all routing. runNamespacedCli() dispatches to ad/pd/tw/migrate/config based on the first positional arg. runCli() handles single-namespace lookup for passthrough modules like migrate. Both parse global flags (--env, --llm) before calling the command module.

command.ts — Module Interface & Result Types

Every command implements CommandModule<TArgs, TData>. Results are wrapped in a CommandEnvelope that includes timing, environment name, schema version, and a success/error discriminator.

interface CommandModule<TArgs, TData> { schema?: string; // e.g. "ad.show" version?: string; // default "2.0" parseArgs?(rawArgs, ctx): TArgs; execute(args, ctx): Promise<CommandResult<TData>>; presentHuman?(envelope, ui): void; // optional custom renderer } // Helper factories: ok(data, meta?) → { ok: true, data, meta } fail(code, msg) → { ok: false, error: { code, message } }

output.ts — Rendering

renderHuman(envelope, presenter?) — If the command exports a custom presentHuman, it is called with the full envelope and a HumanUi helper (sections, tables, key-value pairs, color codes). Otherwise the generic renderer auto-detects: arrays of objects become tables; plain objects become key-value tables; scalars print as text.

renderLLM(envelope) — JSON-stringifies the entire CommandEnvelope to stdout. Agents consume this directly.

Parser (lib/parser.ts)

parseMethodResponse(raw) takes a RawMethodResponse and returns a HydratedMethod. The core challenge is that jsonDefinition is a doubly-serialized JSON string — the parser unwraps it, extracts the inner object, and maps fields to the typed domain model. It also normalises the method name (inner name takes precedence over aliasName) and computes fetchedAt.

Hydrator (lib/hydrator.ts)

ServiceHydrator fetches and caches AutomationDefinition objects for each service in a method chain. Core service IDs (hyphenated, e.g., core-send-email) are cached permanently; all others expire in 1 hour.

Two helper methods extract typed input metadata: findInputLabel(def, inputId) does a recursive search through nested input groups; flattenInputs(def) returns a flat list with depth and ordering for table display.

Method Finder (lib/method-finder.ts)

Builds a full searchable index of all methods and categories by paging through:

GET /rest/api/automation/services?limit=2000&offset=X → categories
GET /rest/api/automation/methods?categoryUuid=X&limit=100&offset=Y → methods per category

Index cached for 7 days. searchMethodIndex(index, query, limit) uses a weighted multi-signal scorer: exact match (1000) → starts-with (900) → word-boundary (850) → contains (760) → subsequence fuzzy (640). Field weights: methodName 1.0, aliasName 0.95, categoryName 0.85.

SQL Engine (lib/sql/)

Executes SQL against NSM databases by injecting queries into special method templates. Five operation types are supported:

FunctionSQL TypeReturn
executeSqlReadQuery()SELECTTyped row array
executeSqlUpdate()UPDATERows affected
executeSqlInsert()INSERTRows affected + generated values
executeSqlModify()DDLStatus

The TUI REPL (lib/sql/tui/) provides an interactive session with database/table selector, query history persistence, and formatted result tables.

Analyzer (lib/analyzer.ts)

analyzeItem(id, type, name, visited, whitelist) is the recursive tree walker. It:

1. Checks the visited Set for cycle detection.
2. Fetches the config for the page/component (from cache).
3. Calls extractReferences() to get AD IDs and child component names.
4. Recursively calls analyzeItem() on each child (in parallel via Promise.all).
5. Aggregates all AD IDs from the subtree into a ReportNode.

Migrations

Source module at migrations/lib/. Orchestrates NSM cross-environment migrations (AD methods, PD pages) by driving an external migration tool via FormData HTTP + WebSocket log streaming.

External Tool Integration

The migration tool is a separate web service at https://db-migration-tool.services.stage.expertly.cloud/. Belz does not have API keys — it acts as an authenticated browser session, bootstrapping cookies via GET /index.html before submitting FormData.

client.ts — Execution Bootstrap

startMigrationExecution(input) submits a FormData payload to POST /executions/start. The form fields map directly to the migration tool's UI inputs:

Form FieldValue
__script_nameMigration script identifier
Choose a profileNSM profile string (e.g., nsmdevncdns_nsmqancdns)
Module NameAD or PD
List of UUIDsComma-separated method/page IDs
Use CRUD API?Boolean toggle

cleanupMigrationExecution(executionId) calls POST /executions/cleanup/{executionId} after streaming completes.

ws.ts — WebSocket Log Streaming

streamMigrationExecution(executionId, options) connects to ws://.../executions/io/{executionId}. On connect, it immediately sends "yes" to confirm the execution. It then collects all incoming text events and typed events until the socket closes (or a 10-minute timeout fires). Returns StreamExecutionResult { executionId, events[], outputText, closeCode, closeReason }.

profiles.ts — Profile Discovery

discoverNsmProfiles() discovers available source→target environment pairs by scraping the migration tool's HTML and bundled JS assets. It searches for strings matching the pattern [a-z0-9]+ncdns_[a-z0-9]+ncdns (e.g., nsmdevncdns_nsmqancdns).

Resolution order: live scrape → cached profiles → hardcoded fallback. Results are cached permanently to ~/.belz/migrations/nsm-profiles.json.

log-parser.ts — Output Parsing

parseMigrationOutput(rawOutput) strips ANSI escape codes, then scans the text for:

  • • Success/failure keywords to set successDetected / failureDetected
  • • URLs for status and details pages
  • • Run ID and migration ID
  • • A JSON block after a known marker line → parsed into reportSummary (entity counts, mismatch counts)

Browser Extension

Architecture

All features are implemented as independent modules under src/features/. Each feature exports an init() function called by the top-level entry point. Features communicate via DOM events and direct DOM manipulation — no background service worker or message passing is used.

Install Script Details (scripts/install.sh)

1. Installs npm dependencies (bun install).
2. Runs generate to rebuild registries.
3. Compiles the binary via bun build --compile.
4. Copies binary to ~/.local/bin/belz.
5. If no ~/.belz/config.json exists: prompts for credentials (or reads from --env-file) and writes the config.

Accepts flags: --env-file <path> (skip interactive prompts), --install-dir <dir> (override binary destination), --yes (reinstall only — skip confirmation).
